# EduForge Codex 指令生成规则 v1.0（精简稳定版）

> 适用于 EduForge 全项目（frontend + backend）的所有 Codex 修改类指令。
> 目标：保证指令结构稳定、输出可审核、执行可规模化。

---

## 1. 固定前缀规则（必须存在）

所有涉及代码修改的 Codex 指令，必须以如下前缀开头：

```text
本指令中：
- 若涉及修改 backend\ 目录下的代码，严格遵循 docs/backend-architecture.md 与 docs/codex-rules.md 的规范。
- 若涉及修改 frontend\ 目录下的代码，严格遵循 docs/frontend-architecture.md 与 docs/codex-rules.md 的规范。
- 若同时涉及修改 backend\ 目录和 frontend\ 目录下的代码，必须分别遵循对应架构规范。
```

此段为强制执行边界声明，不得省略。

---

## 2. 标准结构骨架（栏目固定）

每条 Codex 指令必须包含以下栏目，栏目名称固定：

1. 【任务名】
2. 【任务目标】
3. 【实现范围】
4. 【接口与口径约束】（如无则写“无”）
5. 【强约束（必须遵守）】
6. 【非目标（明确不做）】
7. 【环境变量 / 配置项】（如无则写“无”）
8. 【实现步骤（建议按顺序）】
9. 【预期文件清单（新增/修改）】
10. 【验收步骤（由开发者本地执行）】
11. 【输出要求（本次任务专用，用于审核）】

---

## 3. 核心执行原则

### 3.1 骨架固定，内容弹性

* 栏目结构必须稳定。
* 栏目内部内容可根据任务类型扩展。
* 不允许新增随意的一级栏目。
* 不得删减固定栏目。

---

### 3.2 输出要求必须与任务绑定（关键原则）

“输出要求”不得使用固定模板，必须根据本次任务的风险点和审核重点定制。

输出要求至少包含：

* 新增/修改文件清单
* 本任务的关键实现说明或关键代码片段（视任务类型而定）
* 明确声明未涉及 backend 或未新增依赖（如适用）

输出要求是本规则中最重要的审核锚点。

---

## 4. 不允许的行为

* 不得绕过既有架构规范（backend / frontend 架构文档）
* 不得擅自新增第三方依赖
* 不得跨目录越界修改（除非任务明确允许）
* 不得要求输出每个文件的完整最终内容

---

## 5. 使用方式

在任意新会话中，如需生成 Codex 指令，可直接说明：

> “按本项目文档中的 codex-instruction-spec.md 生成 Codex 指令。”

ChatGPT 必须按本规则输出。

---

## 6. 版本说明

当前版本：v1.0（精简稳定版）
适用阶段：单人主线推进 + 架构已稳定阶段
升级条件：当项目复杂度显著提升时再讨论规则升级
